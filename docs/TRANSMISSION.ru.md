# Руководство по настройке Transmission

Это руководство объясняет, как безопасно настроить Transmission для подключения Transmitter.

## Как Transmitter подключается

Transmitter общается с Transmission исключительно на **стороне сервера** — учётные данные никогда не попадают в браузер. Go-бэкенд аутентифицируется через `Authorization: Basic <base64>` при каждом RPC-запросе и прозрачно управляет токеном `X-Transmission-Session-Id`.

```
Браузер → POST /api/rpc → Сервис Transmitter (аутентификация + белый список) → Transmission RPC
```

Прокси применяет **белый список методов**: пропускаются только `torrent-get`, `torrent-add`, `torrent-start`, `torrent-stop`, `torrent-remove` и `session-get`. Все остальные методы отклоняются с кодом `403`.

## Обязательно: включите аутентификацию RPC

Transmitter **требует** установки `TRANSMISSION_USER` и `TRANSMISSION_PASS`. Соответствующие настройки Transmission должны совпадать.

В файле `/etc/transmission-daemon/settings.json` (остановите демон перед редактированием):

```json
{
  "rpc-authentication-required": true,
  "rpc-username": "ваш-логин",
  "rpc-password": "ваш-пароль"
}
```

Transmission хэширует пароль при первом запуске — введённый вами открытый текст будет заменён хэшем вида `{5d1a45db...`. Это ожидаемое поведение.

Укажите те же учётные данные в `.env`:

```env
TRANSMISSION_USER=ваш-логин
TRANSMISSION_PASS=ваш-пароль
```

## Опционально: белый список IP

`rpc-bind-address: "0.0.0.0"` задаётся намеренно, если вы также используете встроенный веб-интерфейс Transmission с других машин в локальной сети. Аутентификация (описана выше) является основной защитой в этом случае.

В качестве дополнительного уровня можно ограничить IP-адреса, которым разрешено подключаться:

```json
{
  "rpc-whitelist-enabled": true,
  "rpc-whitelist": "127.0.0.1,192.168.88.*"
}
```

Маски поддерживаются по каждому октету. Настройте подсеть под свою локальную сеть.

## Примечание о Docker

Docker Compose Transmitter использует `network_mode: host`, поэтому контейнер разделяет сетевой стек с хостом. `TRANSMISSION_URL=http://localhost:9091/transmission/rpc` работает без изменений независимо от `rpc-bind-address`.

## Рекомендуемые изменения settings.json

Отправная точка — пример конфига, минимально необходимые изменения:

| Настройка | Текущее | Рекомендуемое |
|---|---|---|
| `rpc-authentication-required` | `false` | `true` |
| `rpc-username` | `""` | ваш логин |
| `rpc-password` | `""` | ваш пароль (открытый текст, хэшируется при перезапуске) |

## Проверка подключения

После применения изменений перезапустите Transmission и проверьте эндпоинт health Transmitter:

```sh
curl http://localhost:8080/api/health
# {"status":"ok"}
```

Ответ `502` означает, что Transmitter не может достучаться до Transmission — проверьте `TRANSMISSION_URL`, учётные данные и убедитесь, что Transmission запущен. Ответ `401` от Transmission означает несоответствие учётных данных.

## Не открывайте Transmission RPC во внешнюю сеть

Порт `9091` никогда не должен быть открыт в интернет. Transmitter — единственный предполагаемый клиент: он проксирует, валидирует и аутентифицирует все запросы. Если нужен удалённый доступ, откройте Transmitter (порт `8080`) за обратным прокси с TLS, но не Transmission напрямую.
